**Дата:** 15 декабря 2025 г.

**Статус:** Принято

**Контекст:**

Проект разрабатывается командой 8-12 человек. Необходимо:
- Быстрая поставка новых функций
- Гибкая реакция на инциденты
- Прозрачность статуса разработки
- Минимизация совещаний и документации
- Регулярные выпуски в production

**Рассмотренные варианты:**

1. **Waterfall** – последовательные фазы
2. **Scrum** – фиксированные спринты
3. **Kanban** – непрерывный поток
4. **Scrumban** – гибрид Scrum и Kanban

**Решение:**

Выбран **Scrum с элементами Kanban:** двухнедельные спринты для плановых задач, гибкий поток для инцидентов.

**Обоснование:**

- **Scrum элементы:** планирование, стендапы, обзоры, ретроспективы
- **Kanban элементы:** доска, ограничения по задачам, отдельная дорожка для инцидентов
- **Гибкость для микросервисов:** параллельная работа, независимые выпуски
- **Инструменты:** Jira, GitHub Projects, Slack, Pull Requests

**Недостатки решения:**

- **Риск расползания объёма:** без контроля спринт может перегрузиться
- **Зависимость от планирования:** плохая оценка ведёт к срыву сроков
- **Время на совещания:** требуется несколько часов в неделю
- **Сложность приоритизации:** инциденты vs плановая разработка

**Последствия:**

- Чёткое определение критериев готовности
- Дисциплина в оценке задач
- Доступность Product Owner
- SLA для инцидентов
- Регулярные ретроспективы с улучшениями
- Мониторинг метрик процесса
