**Дата:** 14 декабря 2025 г.

**Статус:** Принято

**Контекст:**

Веб-приложение системы учёта энергопотребления должно:
- Работать в современных браузерах
- Поддерживать обновление данных в реальном времени
- Быть адаптивным и удобным для мобильных устройств
- Быстро загружаться и обеспечивать отзывчивый интерфейс
- Интегрироваться с API бэкенда
- Поддерживать многоязычный интерфейс
- Легко тестироваться и поддерживаться

**Рассмотренные варианты:**

1. **React** – популярная библиотека с большой экосистемой
2. **Vue.js** – прогрессивный фреймворк с отличной документацией
3. **Angular** – полнофункциональный фреймворк со строгой структурой
4. **Next.js** – фреймворк на основе React с серверным рендерингом
5. **Svelte** – компилируемый фреймворк с высокой производительностью

**Решение:**

Выбран **React + TypeScript**.

**Обоснование:**

- **React популярен и зрел:** большое сообщество, множество готовых компонентов
- **TypeScript:** обеспечивает статическую типизацию
- **Next.js для структуры:** встроенный роутинг, серверный рендеринг, оптимизация
- **TanStack Query:** управление серверным состоянием, кэширование
- **Zustand или Jotai для локального состояния:** минимальный шаблонный код
- **Recharts для аналитики:** библиотека для построения графиков
- **Tailwind CSS:** современный подход к стилизации
- **Mobile-first:** адаптивный дизайн
- **Отличный опыт разработки:** горячая перезагрузка, быстрый цикл разработки

**Недостатки решения:**

- **Размер сборки:** требует оптимизации
- **Кривая обучения:** требуется время на освоение
- **Множество зависимостей:** может усложнять поддержку
- **Усталость от обновлений:** требует постоянного обновления зависимостей

**Последствия:**

- Необходимо обновление и мониторинг уязвимостей
- Требуется стандартизация: линтер, форматирование, коммиты
- Настройка CI/CD: проверки, тесты, сборка, развёртывание
- Обучение команды паттернам React
- Стратегия уменьшения размера сборки
